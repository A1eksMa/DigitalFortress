# Требования к хостингу Entry-нод

**Версия:** 1.0
**Дата:** 2025-12-10
**Статус:** Этап 1 (MVP)

---

## 1. Введение

Данный документ определяет требования к выбору хостинг-провайдера для размещения Entry-нод сети. Требования сформулированы с учётом архитектуры и текущего этапа развития проекта (MVP для личного использования).

---

## 2. Контекст

### 2.1. Текущая инфраструктура

- **Core-нода:** Размещена в Японии у проверенного хостера с предоплаченной кредитной линией
- **Домен:** Имеется в наличии
- **Целевая аудитория:** 5-10 пользователей (личное/частное использование)
- **Профиль нагрузки:** Веб-сёрфинг, просмотр мультимедиа

### 2.2. Роль Entry-ноды

Entry-нода — узел первой линии, который:
- Принимает клиентские подключения по VLESS/XHTTP
- Маскируется под легитимный HTTPS-сайт
- Передаёт трафик на Core-ноду через TLS-туннель

## 3. Требования к хостингу

### 3.1. География и юрисдикция

| Параметр | Требование | Обоснование |
|----------|------------|-------------|
| **Приоритетный регион** | Япония | Минимальная задержка до Core-ноды, расположенной в Японии |
| **Альтернативные регионы** | Сингапур, Гонконг, Тайвань, Южная Корея | Юго-Восточная Азия как запасной вариант при отсутствии подходящих японских предложений |
| **Исключённые регионы** | США, Европа, Китай | США/ЕС — приоритетные цели регулятора-цензора; Китай — ограниченный интернет, риски для VPN-сервисов |

**Принцип:** Хостер должен находиться в юрисдикции со свободным интернетом и не подпадать под первоочередные ограничения целевого регулятора.

### 3.2. Тип услуги

| Параметр | Требование | Обоснование |
|----------|------------|-------------|
| **Тип хостинга** | VPS / VDS | Полный контроль над сервером |
| **Root-доступ** | Обязателен | Установка Xray-core, настройка сети и firewall |
| **Выбор ОС** | Обязателен | Требуется Debian или Ubuntu |
| **Контроль firewall** | Обязателен | Настройка iptables/nftables для безопасности |

**Исключаются:**
- Managed-хостинг (WordPress, cPanel-based)
- PaaS-платформы без доступа к ОС
- Shared-хостинг

### 3.3. Сетевые требования

| Параметр | Требование | Обоснование |
|----------|------------|-------------|
| **IPv4** | Выделенный белый адрес | Делегирование домена (A-запись) |
| **IPv6** | Желателен, не обязателен | Дополнительная гибкость |
| **Трафик** | 1-2 TB / месяц | Достаточно для 5-10 пользователей с мультимедиа |
| **DDoS-защита** | Не приоритет | На этапе MVP атаки маловероятны |

### 3.4. Аппаратные ресурсы

| Параметр | Минимум | Обоснование |
|----------|---------|-------------|
| **vCPU** | 1 ядро | Xray-core не требователен к CPU |
| **RAM** | 1 GB | Достаточно для ОС + Xray + лёгкий веб-сервер |
| **Диск** | 10 GB | ОС, логи, статический сайт-прикрытие |
| **Тип диска** | SSD (желательно) | Отзывчивость системы |

**Примечание:** Минимальные ресурсы выбраны исходя из профиля нагрузки (5-10 пользователей, веб + видео). При масштабировании требования будут пересмотрены.

### 3.5. Операционная система

| Параметр | Требование | Обоснование |
|----------|------------|-------------|
| **Поддерживаемые ОС** | Debian 11/12, Ubuntu 22.04/24.04 | Знакомый стек, стабильность, широкая поддержка |
| **Альтернативы** | Не рассматриваются | Унификация инфраструктуры |

### 3.6. Биллинг и оплата

| Параметр | Требование | Обоснование |
|----------|------------|-------------|
| **Валюта** | USD или криптовалюта | Удобство расчётов, избежание конвертации |
| **Исключённые валюты** | CNY (юань), JPY (йена) | Неудобство работы с локальными валютами |
| **Модель биллинга** | Почасовой (желательно) | Гибкость: возможность быстро поднять/удалить ноду без переплаты |
| **Альтернатива** | Помесячный | Допустим при отсутствии почасового |
| **Бюджет** | $5-10 / месяц | Стандартная цена минимальных VPS в регионе |

### 3.7. Регистрация и верификация

| Параметр | Требование | Обоснование |
|----------|------------|-------------|
| **KYC** | Допустим | Качество сервиса важнее анонимности регистрации |
| **Способ регистрации** | Email + платёжный метод | Стандартная процедура |

**Примечание:** Анонимность регистрации не является приоритетом на этапе MVP. При масштабировании может потребоваться пересмотр (множественные аккаунты, ротация).

### 3.8. Terms of Service

| Параметр | Требование | Обоснование |
|----------|------------|-------------|
| **VPN/Proxy** | Желательно отсутствие явного запрета | Снижение риска блокировки аккаунта |
| **Приоритет** | Не блокирующий | Большинство хостеров не проверяют активно |

## 4. Критерии, не приоритетные на этапе 1

Следующие критерии будут актуальны на этапах 2-3, но не влияют на выбор хостера для MVP:

| Критерий | Причина отложения |
|----------|-------------------|
| **API для автоматизации** | Ручной деплой достаточен для 1-3 Entry-нод |
| **Масштабируемость** | Не требуется при 5-10 пользователях |
| **Множественные инстансы** | Одной Entry-ноды достаточно для MVP |
| **Terraform/Pulumi поддержка** | Автоматизация — этап 2 |

---

## 5. Чек-лист для оценки хостера

При оценке потенциального хостера использовать следующий чек-лист:

- [ ] Дата-центр в Японии или ЮВА (не Китай)
- [ ] VPS с root-доступом
- [ ] Выделенный IPv4
- [ ] Debian / Ubuntu в списке ОС
- [ ] 1+ vCPU, 1+ GB RAM, 10+ GB диск
- [ ] 1+ TB трафика в месяц
- [ ] Оплата в USD или крипте
- [ ] Цена ≤ $10/месяц
- [ ] Нет явного запрета VPN в ToS (желательно)
- [ ] Почасовой биллинг (желательно)
